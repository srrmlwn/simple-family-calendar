## Screenshot Observations

### Login page (desktop + mobile)
Clean. Logo, tagline "Planning made easy—type it, say it, done.", private beta banner, Google + email login. Good hierarchy. The card layout on desktop is well-proportioned. Mobile is identical in structure and scales cleanly.

**Minor issue:** The private beta amber banner adds visual noise for invited users who already know they have access. After private beta ends it should be removed. Consider making it a one-time dismissible.

### Register page (desktop + mobile)
**Major inconsistency.** The register page is a completely different visual design from the login page:
- No logo
- No brand color
- No tagline
- Plain white background with a stark gray page
- Form fields are a plain stacked group with no visual delight
- The heading says "Sign up for famcal.ai" + subtitle "Sign up for Simple Family Calendar" — the internal product name "Simple Family Calendar" leaks through and looks like a bug

The user flows from the branded, polished login page to a generic signup form that looks like a different product. This is the first conversion step and it's working against the brand.

### Calendar — Week view (desktop, 1440×900)
The layout is: **[Calendar | 7-column week grid] [Chat sidebar 340px wide]**. This is a meaningful improvement from the old bottom-bar NLP.

**Working well:**
- The 7-column week grid is clean and minimal
- Today (Saturday, Feb 28) has a blue column header — clear and correct
- The "—" placeholder in empty cells is subtle without being noisy
- The toolbar row (February 2026 / Week / Month / Year / Today / ← →) is compact and readable
- The chat panel empty state ("Calendar AI" icon + tagline + input bar) is well-designed

**Issues:**
1. **Chat panel is labeled "Calendar AI"** — a generic product-y name. The whole app is famcal.ai, but the sidebar says "Calendar AI" as if it's a separate feature. It should feel like a first-class citizen of the app, not a named sub-feature. Consider: no label at all, or a simple "Ask anything" placeholder at the top.
2. **Import icon (↓ arrow) in the header** has ambiguous affordance — it looks like a download, not "import from Google Calendar." The tooltip would say "Import events" but first glance it looks like "Export/download." Use an icon that implies importing (file-in, calendar-plus) or add a short label.
3. **Empty columns show "—" dashes** but offer no call-to-action to add an event. A new user sees a grid of dashes and a sidebar. The first prompt should be in the calendar itself: something like a centered "No events — try asking the AI ↗" label in the empty week.
4. **No "new chat" button in the sidebar header when empty.** The empty state shows the icon and tagline, but there's no clear "Start a conversation" button — the user has to notice the input bar at the bottom. The input bar's placeholder text cycles through suggestions, which is great, but it's small.

### Calendar — Week view (tablet, 768×1024)
**Critical breakpoint gap.** At 768px:
- The desktop sidebar requires `min-width: 1024px` (from `useMediaQuery('(min-width: 1024px)')`)
- The mobile vaul drawer should appear, but in the live screenshot the input bar is **not visible at all**
- The calendar fills the entire screen — users on iPad Mini (768×1024 portrait) or similar have no visible AI input entry point

The drawer's CSS is `position: fixed inset-x-0 bottom-0` at 120px snap, so it should overlay the calendar. This needs verification on a real device. If the issue is real, it means a significant percentage of mobile/tablet users (iPad landscape is 1024px wide — only portrait would hit this) are missing the core feature.

**Action required:** Test at 768px on a real device. If the drawer isn't visible, either lower the desktop sidebar breakpoint to 768px, or ensure the drawer snap is rendering.

### Calendar — Month view (desktop, 1440×900)
Clean month grid with today (Feb 28) circled in blue. The sidebar shows the empty state.

**Issues:**
1. **No "+" affordance on month cells.** The week view has `+` buttons that appear on column header hover. The month view cells have no add-event affordance — clicking a date in the month view has no visible action. Users have to use the NLP sidebar or the week view. If you keep the month view, it needs click-to-create.
2. **Month cells are very tall when empty.** A month with events would look fine, but for a new user the month grid is a large empty white space. Consider a minimum content height for cells.
3. **The selected view (Month) still uses the same 3-tab switcher as Week/Year.** The tab is visually clear, but active state styling (bold vs. normal weight) is subtle.

### Calendar — Year view (desktop, 1440×900)
This is the most unconventional screen. The year is rendered as a **continuous horizontal scroll of weeks**, with month labels in colored text on the left edge of each month's start row.

**What's good:** Very compact. You can see the entire year in one screen. Events would appear as colored dots/marks.

**Issues:**
1. **Non-standard format.** A user who has used any calendar app (Google, Apple, Outlook) expects year view to be 12 monthly mini-grids. The horizontal-week format looks like a GitHub contribution graph or a Gantt chart. This isn't necessarily wrong, but it needs to earn its place — either commit to this as a differentiator, or offer a traditional 12-month layout.
2. **Month labels are tiny and colored.** "JAN", "FEB" etc. in different colors on the left edge are small (10–11px). At a glance they read as random colored dots rather than navigation landmarks. At 1440×900, this is already hard to parse.
3. **No interaction affordance.** Clicking a date in year view presumably navigates to that week — but there's no visual affordance (no hover state visible in the screenshot, though code shows click handling).
4. **The year (2026) label in the toolbar is just floating text** with Week/Month/Year tabs. It's not clickable for year navigation.

### Settings page (desktop, 1440×900)
Functional but bare. Sections: Timezone, Co-Manager Access, Family Members, Daily Digest Settings (below the fold).

**Issues:**
1. **No back navigation.** The settings page has no "← Back to Calendar" link or breadcrumb. Users rely on the browser back button or re-clicking the logo.
2. **"Simple Family Calendar" text visible** in the Family Members description. Look for internal naming leaks throughout the app.
3. **Settings sections have no visual hierarchy.** All sections are the same visual weight (white card, bold title). A settings page with 4+ sections should use either tabs, a sidebar nav, or clear visual groupings.
4. **The timezone display is a raw IANA identifier** ("America/Los_Angeles"). Users understand "Pacific Time" — the raw identifier reads like a developer-facing string. Display human-readable names ("Pacific Time (US & Canada)") with the IANA value as secondary metadata.
5. **No save button visible at the top.** Users have to scroll to confirm changes or rely on auto-save. If auto-save is used, a subtle "Saved" indicator is needed after changes.

### Onboarding (desktop + tablet, shown on first login)
The onboarding modal shows over the blurred calendar with 3 steps (dots indicator shows step 1 of 3). Step 1 is "Who's in your family?" with a name input and color picker row.

**What's good:**
- 3 steps is much better than the old 5
- The color picker is visual and immediate
- "Skip for now" option is available
- The blurred calendar backdrop gives context ("this is what you're setting up")

**Issues:**
1. **The step is out of order.** The first onboarding question "Who's in your family?" is not the magic moment — it's admin setup. The user has just signed up and doesn't feel the app's value yet. The magic moment is: *type something and watch it appear on the calendar*. Consider making step 1 the NLP demo ("Try asking: Add Emma's soccer practice Thursday at 4pm") and making family setup step 2.
2. **Color picker has 10 options in a single row** on desktop, wrapping to 2 rows on mobile. The row of circles without labels is fine for selecting a color, but there's no indication which color is currently selected (the selected one has a ring but it's subtle against white).
3. **Step 1 label says "Back"** — but it's the first step. "Back" shouldn't appear on step 1 (it could go to a previous welcome step, but there is none apparently). The button should be disabled/hidden on step 1 or the flow should be re-examined.

## What's Working Well (Keep These)

- **Chat sidebar on desktop** — proper 340px sidebar with message thread, event cards, disambiguation UI is a major step forward from the old bottom bar NLP
- **Vaul drawer architecture on mobile** — the snap point approach (collapsed 120px / expanded 60%) is the right interaction model for mobile NLP
- **Rotating placeholder suggestions** in the input bar — teaches the vocabulary without a tutorial
- **Event card design in chat** — colored left strip, icon, title, time, duration, family member chips are rich and well-structured
- **Week view 7-column grid** — good information density, correct visual hierarchy (today highlighted, past events dimmed)
- **"Nothing scheduled" empty state in week view** — subtle and honest, doesn't clutter
- **Onboarding 3-step trim** — better pacing vs. the old 5-step flow
- **Color-coded family members** — the color dots in event cards and the color picker in onboarding are cohesive

## Prioritized Recommendations

### P0 — Fix Now (Broken or Near-Broken)

1. **Verify tablet NLP visibility (768–1023px).** Test the vaul drawer rendering at 768px on a real device. If broken, either lower the desktop sidebar breakpoint to 768px or fix the drawer. This is the core feature — it must be accessible on tablets.

2. **Remove internal product name "Simple Family Calendar"** from the Register page subtitle and anywhere else it appears. The brand is famcal.ai.

### P1 — High Impact, Low Effort

3. **Restyle the Register page** to match the Login page. Wrap it in the same card layout, add the logo and tagline. This is a single conversion step — it should be polished.

4. **Add back navigation to Settings.** A simple "← Calendar" link in the header or at the top of the page content. Without it, users feel stranded.

5. **Fix "Back" button on onboarding step 1.** Hide it on the first step, since there's nothing to go back to.

6. **Replace IANA timezone string** with human-readable name in Settings. Use a library like `Intl.DisplayNames` or a lookup map.

7. **Clarify the import icon.** Change the `↓ download arrow` to an import-specific icon (e.g., `CalendarPlus` from Lucide) or add a text label on desktop. The download icon looks like export, not import.

### P2 — Medium Impact, Worth Doing

8. **Add an empty-state CTA in the week view** when there are no events. Something like: *"No events this week — ask the AI to add one →"* with an arrow pointing toward the chat sidebar.

9. **Add click-to-create affordance in Month view.** Either a `+` button on cell hover (matching week view behavior) or a visual cue on empty cells.

10. **Swap onboarding step order.** Move the NLP demo ("Try it out" step) to step 1. Let users feel the magic moment before asking them to configure family members.

11. **Rename "Calendar AI" sidebar label.** Remove the feature-name label entirely or replace with a prompt like "What do you need?" — avoid naming it as a sub-product.

### P3 — Design Improvements for Later

12. **Year view rethink.** Either commit to the horizontal-week format with better legibility (larger month labels, hover states, interactivity) or offer a traditional 12-month mini-grid as an alternative. The current implementation is innovative but may confuse users.

13. **Settings page navigation.** As more settings are added, the single-scroll page will become unwieldy. Add anchor navigation (sticky sidebar links or section tabs) now before it becomes a problem.

14. **Consider renaming "Calendar AI" → "AI" or just leaning into the NLP-first metaphor.** If the whole app is the AI family calendar, the sidebar shouldn't be a named feature — it should just be *how you talk to the app*.

15. **Design for the first 5 minutes.** A new user with no events sees: 7 empty columns with dashes, a "Calendar AI" sidebar with an icon. This is a cold start problem. Consider a "getting started" prompt that pre-populates the input with a suggested event.

## Metric to Watch

From the Feb 23 review — this remains the core question:

> **What % of event creations happen via NLP vs. clicking on the calendar?**

Target: >70% via NLP. Every change should be evaluated against this metric.

The chat sidebar redesign is a meaningful step toward this goal. The remaining gap is at the cold start: a new user who sees an empty calendar and an empty sidebar has no strong signal that the right move is to type in the input bar.
