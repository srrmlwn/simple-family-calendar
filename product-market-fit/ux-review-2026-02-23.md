# UX Review — famcal.ai
**Date:** 2026-02-23
**Reviewer:** Claude Code (based on full codebase walkthrough)

---

## Summary

famcal.ai has the right core idea — natural language as the primary interface for a family calendar. But the current UI is fighting itself: it presents a traditional calendar (month grid + day view) as the main visual, then buries its differentiating NLP input at the bottom as a secondary element. The result is an app that looks like every other calendar, with a chat bar tacked on.

The opportunity: lean fully into the NLP-first paradigm. Less calendar chrome, more conversation. Simpler, faster, more memorable.

---

## The Google Principle — One Thing to Look At

Google's homepage is the canonical example of focus: one box, one purpose. Users knew immediately what to do. Everything else was a distraction.

**The question for famcal.ai:** What if the NLP input was the hero, not an afterthought?

### What this could look like

**Current layout hierarchy:**
```
[Header bar]
[Family member filter pills]
[Month calendar | Day view side by side]
[NLP input — bottom bar, visually minor]
```

**Proposed alternative hierarchy:**
```
[Minimal header — logo + profile only]
[NLP input — center, large, prominent]
[Calendar context — appears below/after interaction]
```

On desktop, before you type anything: a large centered input box with a soft hint like *"Schedule something, ask a question, or say what's happening this week"*. Below it, a quiet display of the next 3–5 events. The calendar lives one tap/click away, not as the default view.

On mobile: the NLP bar IS the home screen. Scroll down to see upcoming events. Tap a date chip to drill in.

This reframes the product: not "a calendar with a chat bar" but "an AI assistant that keeps your family schedule."

---

## Current UX Problems to Fix

### 1. Mixed signals about the primary interface

The app has two competing metaphors:
- **Traditional calendar** (click a date, click again to open EventForm, fill in fields)
- **NLP conversation** (type natural language, it handles everything)

Users who discover the traditional click path never learn about NLP. Users who only use NLP never need the calendar view but have to look at it anyway. Commit to one mental model. NLP is the answer — the traditional form should be an escape hatch, not a co-equal path.

**Fix:** Hide the EventForm "click date twice to create" pattern. Let NLP be the only creation method. Keep the EventForm for *editing* existing events.

---

### 2. The NLP input is visually minor

The current bar uses a light indigo background (`bg-indigo-50`) and sits at the bottom of the page — exactly where users look last. The mic and send buttons are small. The placeholder text is functional but clinical: *"Add, update, delete, or ask about events…"*

This is the app's superpower, and it looks like an afterthought.

**Fix options (pick one or combine):**

**Option A — Bigger bar, center-anchored on desktop**
On desktop (before first interaction), show the input in the vertical center of the screen. After first use, animate it to the bottom. Like Spotlight on Mac or the Arc browser command bar.

**Option B — Persistent hero bar at the top**
Move NLP input to the top of the page, full width, replacing the current header. Hint text rotates through examples: *"Add Emma's soccer practice Thursday 5pm"* → *"What's on our schedule this weekend?"* → *"Move dentist to next Tuesday"*

**Option C — Home screen that is only the NLP bar**
First screen after login is just the input + a few recent events below it (like a personal Perplexity). Calendar view lives behind a "View calendar" button. This is the boldest, highest-conviction version of the NLP-first bet.

---

### 3. The month calendar view is rarely the right view

A monthly grid is useful for planning ahead and getting a bird's-eye view. But busy parents checking "what do we have today?" or "what's this week?" don't need a month view — they need a fast answer.

**Observation:** The DayView (right panel) is actually very useful. The month grid (left panel) on desktop mainly serves as a date-picker for the DayView.

**Fix:** Replace the month grid with a **7-day "this week" agenda** as the default view. Keep the month mini-calendar as a collapsible nav element (like Google Calendar's left sidebar). This shows more *content* (actual events) with less *chrome* (grid squares).

The NLP can answer "what's on the 15th?" without the user needing to click to it in a month grid.

---

### 4. Family member filter takes up real estate it doesn't earn

The colored pills at the top of the calendar (`FamilyMemberFilter`) are visually noisy on first load. Most of the time, parents want to see all events. Filtering by one member is an occasional operation.

**Fix:** Move family member filtering into the NLP ("show me just Emma's events this week"). Remove the persistent pill row from the main view. This cleans up 40–60px of vertical space on mobile.

Alternatively, make filter pills collapsible/hidden by default, expandable with a "Filter" button or chevron.

---

### 5. Two-click pattern to create events is non-obvious

Currently: click a calendar date once to select it, click it again to open the event form. This is invisible — there's no visual affordance that a second click will do something.

Most users will try to type in the NLP bar first (which is correct behavior!), but then be confused if they want to use the calendar directly.

**Fix:** If keeping click-to-create, use a `+` button on day cells in the day view (like Google Calendar). Or remove it entirely and rely on NLP only for creation.

---

### 6. "Click to edit" in EventForm is not universal UX

The inline click-to-edit pattern with pencil icon on hover is used in EventForm. This works for power users but can confuse casual users who don't know fields are editable.

**Fix:** Use standard input fields (always visible, clearly inputs). The current pattern optimizes for read/display mode, but an event form is inherently in *edit mode*. Just show the fields as inputs from the start.

---

### 7. Onboarding is 5 steps — too long for a calendar app

The 5-step onboarding wizard covers: Welcome → Family Members → Notifications → Email Recipients → Try It Out. Step 4 (email recipients) is an advanced feature that most users won't need on day one.

**Fix:** Cut to 2 steps:
1. **Add your family** (names + colors) — this personalizes the app immediately and creates the "your family" feeling
2. **Try it out** (pre-filled NLP example, let them send it, see the event appear)

That's it. Notifications and email recipients can be discovered through Settings later. The sooner users experience the NLP "magic moment," the better.

---

### 8. Placeholder text doesn't teach the capability

Current placeholder: *"Add, update, delete, or ask about events…"*

This is accurate but it doesn't inspire or demonstrate. A user who has never used NLP calendar input doesn't know what "ask about events" even means.

**Fix:** Use rotating example placeholders or a static but concrete example:
- *"Add dentist Tuesday at 3pm for the whole family"*
- *"Move soccer practice to Thursday"*
- *"What does Emma have this week?"*
- *"Cancel Friday's swimming lesson"*

Each example teaches a capability. Rotate them on a timer, or show a different one each session.

---

### 9. Toast notifications disappear too fast for first-time users

Success toasts ("Created 'Dentist Appointment'") dismiss after 2.5 seconds. For someone who just used NLP for the first time and wants to verify it worked, 2.5 seconds is not enough time to read and feel confident.

**Fix:** Extend to 4 seconds. Or replace toasts with a persistent "last action" indicator that shows the last thing the AI did, until the next action replaces it. This also helps with the "did that work?" anxiety.

---

### 10. Header is generic, not brand-building

The current header shows the famcal.ai logo (text), a settings gear, and a user avatar. It's functional but could do more work.

**Fix:** Add a subtle value reminder — something like "Your family's AI calendar" in the header, or a daily greeting ("Good morning, Sriram — 3 events today"). This reinforces the product's character on every visit.

## Medium Bets (Worth a Design Spike)

These require more thought and may need A/B testing or user feedback:

| Idea | Hypothesis | Risk |
|------|-----------|------|
| **NLP-first home screen** (no calendar on landing, just the input + next few events) | Reinforces the product identity and NLP-first mental model | Might disorient users expecting a calendar |
| **Replace month grid with 7-day agenda** as default view | More useful information density for busy parents | Loses planning/overview capability |
| **Move NLP input to center/top on desktop** | Signals importance, drives adoption | Layout shift may feel jarring |
| **"Today's briefing" default view** (AI-generated summary of today + this week) | Delivers immediate value without any user action | Requires reliable NLP + potential for wrong info |
| **Collapse all traditional calendar chrome by default** (hide until explicitly requested) | Forces NLP adoption, cleaner first experience | Alienates users who want traditional calendar interface |

## Bolder Vision — What This Could Be

The boldest version of famcal.ai isn't "calendar with NLP." It's closer to:

> "Ask it anything about your family's schedule. It just knows."

**Imagine the first screen after login:**

```
┌─────────────────────────────────────────────────────────┐
│  famcal.ai                                    [profile] │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Good morning, Sriram.                                 │
│   Emma has soccer at 4pm. Dentist tomorrow at 3pm.      │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  What's happening this week?               🎤 ▶ │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  [This Week ↓]  [Calendar View]  [Settings]             │
│                                                         │
│  ──── Today, Mon Feb 23 ────                            │
│  4:00pm  Soccer practice  · Emma                        │
│                                                         │
│  ──── Tomorrow, Tue Feb 24 ────                         │
│  3:00pm  Dentist appointment  · Everyone                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

The calendar view is one tap away. The NLP input is the home. The AI greets you with what it knows. This is a fundamentally different product from Google Calendar — it has a personality and a point of view.

## What to NOT Change

Not everything needs to change. These are working well and should be preserved:

- **BottomSheet pattern** — sliding up on mobile, centered modal on desktop. Feels right for this app.
- **Family member color coding** — visible, quick, and functional. Dot indicators on event cards are subtle and correct.
- **Voice input with visual feedback** — the pulsing red mic button is a good affordance. Keep it.
- **Inline editing in EventDetails** — once you're IN the event, the click-to-edit pattern is fine (you're already in "manage mode").
- **Recurring event scope dialog** — the 3-option pattern (this / future / all) is the right UX for this problem.
- **Split day view** — showing time + title + location + family member dots is information-dense in the right way.

## Key Metric to Watch

The core UX question for this product is:

> **What percentage of event creations happen via NLP vs. clicking on the calendar?**

If that number is below 70%, the app is not delivering on its core premise. Every UX change should be evaluated against whether it moves that number up.
